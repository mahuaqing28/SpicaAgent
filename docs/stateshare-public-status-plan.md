# stateShare Public Status Metadata Plan

This document describes the expected app and SpicaAgent changes if
`time-is-money-app` can edit the public `tagline`, `funnyStatus`, and
`today_bgm` values that appear on the `stateShare` GitHub Pages site.

No implementation is included here. The goal is to define a clean contract and
keep `stateShare/data/status.json` generated by the remote SpicaAgent server.

## Target Public Payload

SpicaAgent should continue writing `stateShare/data/status.json`, but with the
new stateShare shape:

```json
{
  "owner": "2049",
  "today": "2026-06-06",
  "updatedAt": "2026-06-06 20:30",
  "tagline": "把今天排成能真实发生的形状。",
  "funnyStatus": "今日系统在线，准备按计划推进。",
  "today_bgm": "Night Drive - 2049 Radio",
  "schedule": [
    {
      "date": "2026-06-06",
      "title": "深度工作",
      "type": "TIME_BLOCK",
      "start_time": "14:00",
      "end_time": "16:00"
    },
    {
      "date": "2026-06-06",
      "title": "跑步准备体测",
      "type": "FLOATING"
    }
  ]
}
```

Notes:

- `owner` defaults to `2049`.
- `schedule[].status` is not included.
- `schedule[].note` is not included.
- `start_time` and `end_time` are optional display strings derived from
  `start_time_ms` and `end_time_ms`.
- `TIME_BLOCK` entries are shown on the timeline by stateShare.
- `FLOATING` entries are shown as non-timeline floating arrangements.

## App Changes

The Android app should still submit schedule facts to SpicaAgent. It should not
write Git files and should not generate `status.json` itself.

Add editable public status fields in app storage:

```kotlin
data class SpicaPublicStatusSettings(
    val tagline: String = "",
    val funnyStatus: String = "",
    val todayBgm: String = ""
)
```

Storage can live beside the existing Spica bridge settings, or in a separate
DataStore/SharedPreferences namespace. Empty values should mean "use the server
default".

Add UI controls where the current Spica sync settings live:

- `tagline`: short text input.
- `funnyStatus`: short text input.
- `today_bgm`: short text input.
- Save action persists the values locally.
- Sync now sends the latest values together with the schedule snapshot.

Extend the schedule snapshot payload with a nested object:

```json
{
  "device_id": "...",
  "sent_at_ms": 1710000000000,
  "timezone": "Asia/Shanghai",
  "today": "2026-06-06",
  "source": "time-is-money-app",
  "state_share": {
    "tagline": "把今天排成能真实发生的形状。",
    "funny_status": "今日系统在线，准备按计划推进。",
    "today_bgm": "Night Drive - 2049 Radio"
  },
  "tasks": [],
  "schedules": []
}
```

Use a nested `state_share` object instead of new top-level fields so that public
presentation metadata remains separate from schedule data.

For `POST /api/schedule/changes`, either include the latest `state_share` object
every time or omit it when unchanged. The server should support both. Including
it every time is simpler and keeps the remote state fresh after incremental
syncs.

Recommended validation on the app side:

- Trim whitespace.
- Limit each value to a modest length, for example 120 characters.
- Allow empty values.
- Do not block schedule sync if these fields are empty.

## SpicaAgent Server Changes

SpicaAgent should remain the source of truth for public `status.json`
generation. The app only supplies optional public metadata.

### Configuration

Add defaults:

```env
SCHEDULE_STATESHARE_OWNER=2049
SCHEDULE_STATESHARE_DEFAULT_TAGLINE=
SCHEDULE_STATESHARE_DEFAULT_FUNNY_STATUS=
SCHEDULE_STATESHARE_DEFAULT_BGM=
SCHEDULE_STATESHARE_AUTO_COMMIT=false
SCHEDULE_STATESHARE_REPO=/home/mahuaqing/personnalProject/stateShare
SCHEDULE_STATESHARE_PUSH=true
SCHEDULE_STATESHARE_BRANCH=main
SCHEDULE_STATESHARE_REMOTE=origin
SCHEDULE_STATESHARE_COMMIT_TIMEOUT_SECONDS=30
```

Existing `SCHEDULE_STATESHARE_FILE` should continue to point at:

```text
/home/mahuaqing/personnalProject/stateShare/data/status.json
```

### Parsing

Add an optional parser for `payload["state_share"]`:

```python
@dataclass(frozen=True)
class StateShareMetadata:
    tagline: str
    funny_status: str
    today_bgm: str
```

Parsing rules:

- Missing `state_share` is valid.
- Non-object `state_share` should reject the payload with `SchedulePayloadError`.
- Values are optional strings.
- Trim values.
- Enforce length limits server-side too.

### Persistence

Persist the latest public metadata in the schedule state file, for example:

```json
{
  "state_share": {
    "tagline": "...",
    "funny_status": "...",
    "today_bgm": "..."
  }
}
```

This prevents metadata from disappearing when the server restarts or when
phone-status-only updates refresh the public file.

Update metadata when:

- `process_snapshot()` receives `state_share`.
- `process_changes()` receives `state_share`.

Leave existing metadata unchanged when a payload omits `state_share`.

### Public Payload Generation

Update `_state_share_payload_locked()` to produce the new stateShare contract:

- Use configured owner, defaulting to `2049`.
- Use app-supplied `tagline`, `funny_status`, and `today_bgm` when present.
- Fall back to configured defaults when app values are empty.
- Iterate over `self._sorted_schedules_locked()`, not `_display_items_locked()`.
- Join schedule entries to their task-derived title using the existing
  `ScheduleEntry` data.
- Emit only public fields:
  - `date`
  - `title`
  - `type`
  - optional `start_time`
  - optional `end_time`
- Do not emit task descriptions.
- Do not emit completion state.
- Do not emit reminder fields.
- Do not emit task ids, schedule ids, package names, phone status details, or
  other private data.

`start_time` and `end_time` should be formatted from milliseconds using the
same timezone context used for schedule display.

### Endpoint Behavior

`GET /api/schedule/stateshare` can keep the same path and authentication model,
but it should return the new payload shape.

`POST /api/schedule/snapshot` and `POST /api/schedule/changes` response bodies
do not need to change. If useful for debugging, they may include a simple field
such as:

```json
{
  "state_share_updated": true
}
```

That is optional.

### Git Commit Automation

Automatic stateShare commits should be handled after file generation, not by the
Android app.

Recommended behavior:

- Write `data/status.json`.
- Check whether the stateShare repo has a diff.
- If no diff, skip commit.
- If changed, run `git add data/status.json`, `git commit`, and `git push`.
- Serialize commits with a lock to avoid overlapping syncs.
- Log git failures without failing the schedule sync request.

This keeps schedule sync reliable even when GitHub or network access is
temporarily unavailable.

## Testing

Add SpicaAgent tests for:

- Snapshot accepts `state_share` metadata.
- Changes accept `state_share` metadata.
- Metadata persists in the state file.
- Missing `state_share` preserves existing metadata.
- Bad `state_share` type is rejected.
- Public stateShare payload uses `owner = "2049"` by default.
- Public payload includes `tagline`, `funnyStatus`, and `today_bgm`.
- Public `schedule` uses only scheduled entries.
- Public `schedule` does not include `status` or `note`.
- `TIME_BLOCK` entries include formatted `start_time` and optional `end_time`.
- `FLOATING` entries omit time fields when none are present.

Add Android tests for:

- Public status settings persistence.
- Snapshot payload includes the nested `state_share` object.
- Empty public fields are allowed.
- Sync still works when public fields are blank.

## Rollout Order

1. Update stateShare front-end to the new `status.json` schema.
2. Update SpicaAgent to parse, persist, and generate the new public payload.
3. Update Android app UI and payload generation.
4. Add optional SpicaAgent git commit automation.
5. Enable auto commit only after generated payloads are stable.
